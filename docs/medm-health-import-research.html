MedM Health Body Weight Import Research

Feasibility review for importing body-weight readings from MedM Health or a MedM-connected scale into Habit Tracker.

Prepared June 7, 2026. Implementation updated June 10, 2026. Worktree: codex/medm-health-import.

Recommendation

MedM body weight ingestion looks feasible for v1 through user-controlled file export, with Apple Health as the best iPhone source of truth if MedM sync is already enabled. Start with CSV/XLSX-style file import for weight readings and avoid undocumented cloud APIs.

Prototype

This branch now includes a source-specific MedM body-weight CSV adapter. It is a local CLI utility only: it performs no Supabase writes, applies no migrations, and builds no dashboard UI.

Run it against a synthetic fixture with:

node scripts/medm-health/parse-medm-weight-csv.mjs scripts/medm-health/fixtures/synthetic-medm-weight.csv

It prints normalized JSON to stdout and row-level warnings to stderr. Malformed rows are skipped with row numbers and compact row context, while valid rows continue through. If the file has no usable header/data rows or no valid body-weight rows, the CLI exits nonzero.

Expected output rows use the shared normalized metric shape:

{
  "date": "2026-06-01",
  "metric_type": "body_weight",
  "value": 181.8,
  "unit": "lb",
  "source": "medm_health",
  "source_detail": "Synthetic Scale; MedM Health CSV; time 07:15",
  "source_record_id": "medm-synth-001"
}

Parser Contract

The parser is adapter-only and performs no database writes. It accepts synthetic MedM-style CSV exports and emits JSON rows shaped as { date, metric_type, value, unit, source, source_detail?, source_record_id?, raw_hash? }.

  • metric_type is always body_weight.
  • source is always medm_health.
  • unit is normalized to lb or kg from the unit column, value text, or weight header.
  • source_detail combines available app, device, export source, and time details.
  • source_record_id is used when an export row has an ID; otherwise raw_hash provides stable dry-run idempotency. Duplicate rows without IDs produce the same hash deterministically.

Supported Synthetic CSV Variants

The parser currently tolerates comma, semicolon, or tab delimiters; quoted fields; a UTF-8 byte-order mark; header variants such as Date, date, Timestamp, Time, Weight, Body Weight, Body Mass, Unit, and Weight Unit; ISO dates; U.S. slash dates; day-month-year dotted dates; separate date/time columns; weight values with embedded units; pounds and kilograms; and decimal commas in semicolon-style exports. Rows missing date, numeric weight, or a supported unit are skipped with warnings.

Verification Fixtures

  • scripts/medm-health/fixtures/synthetic-medm-weight.csv covers common pound and mixed-unit rows with explicit record IDs.
  • scripts/medm-health/fixtures/synthetic-medm-weight-kg.csv covers semicolon-delimited kilogram rows and decimal-comma values.
  • scripts/medm-health/fixtures/synthetic-medm-weight-malformed.csv covers invalid dates, nonnumeric weights, and unsupported units.

Run parser checks with:

node --check scripts/medm-health/parse-medm-weight-csv.mjs
node --check scripts/medm-health/test-parse-medm-weight.mjs
node scripts/medm-health/test-parse-medm-weight.mjs

MedM Health Portal Sync Spike

This branch also includes a local-only Playwright spike for ordering and downloading a MedM Health Portal weight report. It is intentionally dry-run only: it logs report status, CSV discovery, parsed row count, and parsed date range, but does not print body-weight values or write to Supabase.

Discovered Portal Flow

  1. Log in to https://health.medm.com with a MedM user account.
  2. Create a report with POST /en/records/:recordId/reports.
  3. Submit from_date, to_date, medical_record_id, queue_period=never, measurement_types[]=weight, format=csv, and group_by=type.
  4. Read GET /en/records/:recordId/reports and find the newest /download_report?id=... link.
  5. Download the ZIP report, extract CSV files locally, and parse each CSV through the MedM body-weight adapter.

Local Usage

Install root dependencies and run a dry run with credentials supplied by environment variables:

npm install
$env:MEDM_EMAIL="you@example.com"
$env:MEDM_PASSWORD="..."
$env:MEDM_RECORD_ID="..."
npm run medm:sync:dry-run -- --days 30

Downloaded ZIP and CSV files are saved only under local-data/imports/medm-health/. The script does not create Playwright storage-state files, commit cookies, or upload artifacts.

Real Import

Dry-run is the default. A real import requires the Supabase migration 20260610000000_create_daily_metrics.sql to be applied first, then an explicit import command:

$env:SUPABASE_URL="https://..."
$env:SUPABASE_SERVICE_ROLE_KEY="..."
$env:SUPABASE_METRICS_USER_ID="auth-user-uuid"
npm run medm:sync:import -- --days 30

The import maps MedM weight rows into daily_metrics and upserts on user_id, metric_date, metric_type, source. If a report contains multiple MedM weight rows for the same day, the importer collapses them deterministically to one daily row before upsert. It imports body_weight only and does not process unrelated health measurements. The service-role key is for local/server automation only and must never be exposed in client code.

GitHub Actions

The workflow .github/workflows/medm-health-sync.yml supports manual runs and a daily cron. Scheduled runs are dry-run only. Manual runs default to dry-run and can opt into import by setting dry_run=false.

Dry-run requires these repository secrets:

  • MEDM_EMAIL
  • MEDM_PASSWORD
  • MEDM_RECORD_ID

Real import additionally requires:

  • SUPABASE_URL as a GitHub secret
  • SUPABASE_SERVICE_ROLE_KEY as a GitHub secret
  • SUPABASE_METRICS_USER_ID as a GitHub Actions variable containing the target auth.users.id

The workflow installs Playwright Chromium, runs synthetic parser/helper tests, and then runs the portal sync. It does not upload downloaded reports as artifacts and logs only report status, CSV filenames, parsed row counts, and date ranges.

Storage

The new daily_metrics table is generic and private. RLS allows authenticated users to read, insert, and update only rows where auth.uid() = user_id. The automation upsert uses server-side secrets and the target user id; no service-role secret is used in browser/client code.

Rollback and Safety

  • To stop imports, leave the workflow in dry-run mode or remove the Supabase import secrets/variable.
  • To remove imported MedM rows after testing, delete rows where source = 'medm_health' for the target user.
  • Do not commit files from local-data/. Real MedM ZIP/CSV downloads are local-only debug artifacts.

Data Fields Available

For weight, MedM documents Body Mass / Weight plus body composition fields such as body fat percentage, BMI, muscle metrics, water metrics, basal metabolic rate, visceral fat, lean mass, and metabolic age. The first import should only support body_weight, then add BMI/body-fat once a real export format is inspected locally under local-data/.

Manual User Steps

  1. Pair the body-weight scale with MedM Health Diary or enter weight manually.
  2. Enable Apple Health sync for Weight if using Apple Health as the hub.
  3. Export a weight report/file from MedM or Apple Health.
  4. Place real local exports under local-data/imports/medm-health/. The entire local-data/ tree is gitignored and should remain private.
  5. Run a local dry-run importer and review normalized rows before uploading.

Privacy Rules

  • Do not commit raw body-weight exports, health dumps, cookies, tokens, .env.local, Supabase secrets, or personal measurement data.
  • Only synthetic fixtures belong in Git.
  • Warnings may echo compact row context for debugging, so treat CLI output from real files as local/private.
  • Portal sync downloads are local dry-run material and belong only under gitignored local-data/.

Risks and Limitations

  • Actual MedM export headers need to be inspected with a real local file before final parser hardening.
  • Premium subscription may affect available report/export and ecosystem sync options.
  • Body composition fields vary by scale and may not map cleanly across Apple Health, MedM, and Habit Tracker.
  • Do not commit real weight exports; keep them under gitignored local-data/.
