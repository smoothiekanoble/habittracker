This is the living handoff file for the project: what the app is trying to be, what is already working, what blocks deployment, and what path should get a real v1 onto phone and laptop with the least ceremony.

Current Read

Best near-term v1 path Use the Next.js web app as a hosted/PWA-style app on iPhone and laptop. It already uses Supabase as the source of truth, so data survives across devices.
Native iOS status The SwiftUI app exists, but Windows cannot run Xcode/iOS tooling. More importantly, the iOS app has no Google sign-in UI yet, so Supabase sync is effectively dormant until native auth is wired.
Current deployment focus The repo is now on main, pushed to GitHub, and builds from C:\dev\personal\habittracker\web. The remaining v1 work is deployment setup: Vercel project settings, Supabase env vars, and production Auth redirect URLs.

Architecture

LayerWhat It Doesv1 Meaning
web/ Next.js 14 app with Google OAuth through Supabase, habit CRUD, Today view, monthly completion calendar, and mobile web manifest. Primary shippable client.
supabase/ Migrations for habits and habit_logs, with RLS scoped to auth.uid(). Canonical data store.
shared/ TypeScript domain types for habits, logs, and widget snapshot payloads. Useful contract layer; no business logic.
ios/ SwiftUI app, SwiftData local store, widget extension, App Group snapshot contract, Supabase sync service skeleton. Promising, but not ready as the fastest Windows-only route.

OptionWhat Has To Be TrueAssessment
Hosted web app + Add to Home Screen Deploy Next app, configure Supabase URLs, sign in on phone/laptop. Recommended v1. No Mac needed, cloud data already works, and this gets the app onto the phone as an icon-like experience.
Native iOS through TestFlight Apple Developer Program, signed archive from Xcode/Xcode Cloud/CI, App Store Connect setup, native OAuth implemented. Reasonable later. Better than weekly sideloading once native auth exists, but there is setup overhead.
Weekly free sideload A Mac with Xcode or a signing/build service, device connected/registered, rebuild/reinstall when the provisioning profile expires. Not ideal. Apple free personal-team profiles can expire after 7 days, and this still does not solve the missing native auth path.
Public App Store Developer Program, App Store Connect record, signed build upload, app review, privacy metadata, screenshots. Possible without owning a Mac only if a cloud macOS build/upload path is set up, but overkill for a one-user v1.

Decision: Ship v1 as the web app first. Treat the native iOS app/widget as v1.5 once there is either Mac access, Xcode Cloud/GitHub Actions signing, or a managed cloud iOS pipeline.
