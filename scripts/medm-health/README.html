MedM Health Adapter Contract

This directory contains source-adapter work for MedM body-weight CSV exports. It intentionally does not define database schema, apply Supabase migrations, or build application UI.

Command

node scripts/medm-health/parse-medm-weight-csv.mjs scripts/medm-health/fixtures/synthetic-medm-weight.csv

Portal Dry Run

npm install
$env:MEDM_EMAIL="you@example.com"
$env:MEDM_PASSWORD="..."
$env:MEDM_RECORD_ID="..."
npm run medm:sync:dry-run -- --days 30

The portal dry run uses Playwright to order a MedM Health Portal weight CSV report, download the ZIP, extract CSV files under local-data/imports/medm-health/, and parse them without printing body-weight values.

Import

$env:SUPABASE_URL="https://..."
$env:SUPABASE_SERVICE_ROLE_KEY="..."
$env:SUPABASE_METRICS_USER_ID="auth-user-uuid"
npm run medm:sync:import -- --days 30

Import requires the daily_metrics migration to be applied first. It upserts body-weight rows only and remains server/local automation, not client-side code.

Output Shape

{
  "date": "YYYY-MM-DD",
  "metric_type": "body_weight",
  "value": 181.8,
  "unit": "lb",
  "source": "medm_health",
  "source_detail": "Synthetic Scale; MedM Health CSV; time 07:15",
  "source_record_id": "medm-synth-001"
}

Rules

  • Use only synthetic fixtures in Git.
  • Keep real exports under local-data/imports/medm-health/.
  • Use source_record_id when supplied by the export; otherwise use raw_hash for idempotency.
  • Apply the daily_metrics migration before running a real import.
  • GitHub Actions requires MEDM_EMAIL, MEDM_PASSWORD, and MEDM_RECORD_ID secrets for dry-run. Manual import also requires Supabase import secrets and SUPABASE_METRICS_USER_ID.
